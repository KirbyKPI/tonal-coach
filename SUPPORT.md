# Support

Use the right channel so issues get handled efficiently.

## Security

For credential handling, encryption, or other security concerns, follow
[SECURITY.md](SECURITY.md). Do not open a public issue for vulnerabilities.

## Bug reports

Use the GitHub bug report template when:

- the issue is reproducible
- you can describe expected vs actual behavior
- you can share enough detail for someone else to verify it

Include whether you are using the hosted app or a self-hosted deployment, the
environment you tested in, and the smallest reproduction you can provide.

## Feature requests

Use the GitHub feature request template for improvements that fit the current
scope of the project. Read [CONTRIBUTING.md](CONTRIBUTING.md) first so requests
line up with the architecture and roadmap boundaries.

## Self-hosting help

Start with:

- [README.md](README.md) for setup and deployment
- [docs/trust-model.md](docs/trust-model.md) for the data-handling model

If you still believe there is a product bug, open a bug report with logs,
screenshots, and the exact step that fails.

## Response expectations

This is a personal project maintained on a best-effort basis. There is no
guaranteed SLA for general support, issue triage, or feature requests.

## Community

There is a [Discord server](https://discord.gg/Sa5ewWP5M) for casual discussion.
GitHub issues remain the primary channel for bugs, features, and support.

## What support is not

- The issue tracker is not a private help desk
- Maintainer time for custom deployment consulting is not guaranteed
